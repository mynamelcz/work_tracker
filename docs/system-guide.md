# Chip Todo 系统说明文档

## 1. 项目概述

Chip Todo 是一个面向芯片测试团队的任务管理工具，采用单页应用结构，所有数据都保
存在浏览器本地。它适合团队在没有后端服务、没有账户体系的场景下快速搭建统一的任
务看板、项目管理页和会议记录页。

系统设计重点不是复杂权限，而是让三类工作在同一套本地数据上协同：

- 日常查看进度：通过看板完成
- 维护项目和任务：通过管理页完成
- 开会同步进展：通过会议页完成

## 2. 当前界面职责

### 2.1 看板

看板是只读视图。

- 按成员展示任务甘特式卡片
- 支持按状态筛选：全部、待处理、进行中、暂停、已完成
- 支持按项目过滤
- 点击任务打开只读详情
- 详情中可查看任务基础信息、持续时间、会议中的进展记录和阻塞问题

注意：

- 看板不允许直接编辑任务
- 任务的负责人、状态、优先级等结构化信息应在管理页维护
- 任务进度在会议中修改后，会回流到看板显示

### 2.2 管理

管理页负责结构化维护。

- 新建项目
- 编辑项目状态
- 删除项目
- 新建成员
- 编辑成员信息
- 删除成员
- 在项目详情中新增、编辑、删除任务

任务编辑允许维护：

- 任务名称
- 所属项目
- 负责人
- 进度
- 优先级
- 状态
- 描述

### 2.3 会议

会议页负责会议记录和会议过程中的进度同步。

- 新建会议
- 查询会议历史
- 导入参会人员
- 为每个参会人员导入其未完成任务
- 记录进展记录和阻塞问题
- 调整任务进度
- 保存会议
- 导出 HTML 报告

会议页有两个重要约束：

- 它不会承担任务结构编辑职责，例如修改项目归属、负责人、描述等
- 它保存的是会议记录，同时把进度更新回任务主数据

## 3. 页面工作流

## 3.1 日常使用建议

1. 在管理页建立项目、成员、任务
2. 在看板查看当前工作负载和按状态筛选结果
3. 开会前在会议页新建会议并导入参会人员
4. 按人导入未完成任务
5. 会议中记录任务进展和阻塞问题
6. 保存会议并导出 HTML 报告
7. 会后回到看板或管理页查看同步后的最新进度

## 3.2 会议创建与保存流程

会议页的设计逻辑如下：

- 新建会议时会生成一份会议草稿，默认主题为`周会`
- 会议会记录精确创建时间，不再依赖“第几周”的概念
- 草稿中可以导入参会人员
- 对每位参会人员，只能导入该成员未完成的任务
- 未完成任务包含：
  - 进行中
  - 暂停
  - 待处理（如果数据里存在）
- 已完成任务不会进入导入列表
- 保存会议时会要求确认
- 如果当前会议有未保存修改，点击新建会议会先提醒保存当前内容

## 3.3 会议历史与快照

会议记录采用“已保存快照 + 任务回写”的组合模型。

保存会议后：

- 会议标题、创建时间、备注、参会人、任务列表会作为会议记录保存
- 每条任务的`进展记录`和`阻塞问题`会进入`taskReports`
- 任务当前进度会回写到主任务数据

查看历史会议时：

- 优先读取已保存的会议快照
- 不再用当前实时任务覆盖历史内容
- 任务详情里会从所有会议记录中汇总该任务的会议历史，按时间倒序显示

## 4. 数据模型

## 4.1 localStorage Key

- `chip_todo_data`：主数据
- `chip_todo_meetings`：会议记录
- `chip_todo_history`：周数据归档

## 4.2 主数据结构

```js
{
  members: [{ id, name, role, color, createdAt }],
  projects: [{ id, name, description, status, weekKey, createdAt }],
  tasks: [{
    id,
    projectId,
    name,
    description,
    assignee,
    status,
    priority,
    progress,
    weekKey,
    createdAt,
    completedAt,
    pausedAt
  }],
  currentWeek,
  currentYear
}
```

## 4.3 会议记录结构

```js
{
  id,
  title,
  attendees: [memberId],
  notes,
  tasks: [{
    id,
    projectId,
    projectName,
    name,
    assignee,
    assigneeName,
    priority,
    progress,
    status
  }],
  taskReports: {
    [taskId]: {
      work,
      issues,
      plan,
      updatedAt
    }
  },
  createdAt,
  updatedAt
}
```

## 5. 状态与进度规则

### 5.1 任务状态

- `pending`：待处理
- `in_progress`：进行中
- `paused`：暂停
- `completed`：已完成

### 5.2 项目状态

- `in_progress`：进行中
- `paused`：暂停
- `completed`：已结项

### 5.3 优先级

- `low`：低
- `medium`：中
- `high`：高

### 5.4 进度同步规则

- 任务完成时进度强制为`100`
- 进行中或暂停状态下，进度不会被保留为`0`或`100`
- 会议页修改任务进度后，会同步影响：
  - 看板显示
  - 管理页任务列表
  - 后续会议导入时的任务快照

## 6. 代码结构

### 6.1 `chip-todo/js/utils.js`

通用工具模块，负责：

- SVG 图标
- DOM 查询与创建
- 文本转义
- 状态与优先级文案
- 模态框与确认弹窗
- 文件下载

### 6.2 `chip-todo/js/store.js`

数据层，负责：

- 读取和保存本地数据
- 归一化项目、成员、任务、会议数据
- 封装增删改查
- 会议快照和任务回写
- 会议搜索和会议历史汇总

### 6.3 `chip-todo/js/app.js`

主界面逻辑，负责：

- 整体渲染
- 事件绑定
- 三个主视图的交互
- 各类模态框
- 会议 HTML 报告导出

### 6.4 `chip-todo/css/style.css`

单文件样式，负责：

- 页面布局
- 三个视图的组件样式
- 会议页任务条、卡片、表单和历史列表样式
- 移动端适配

## 7. 运行与测试

## 7.1 本地运行

直接打开：

```bash
start chip-todo/index.html
```

或启动静态服务：

```bash
python -m http.server 5500 -d chip-todo
```

## 7.2 自动化测试

安装依赖：

```bash
npm install
npx playwright install chromium
```

运行测试：

```bash
npm test
```

单文件：

```bash
npm run test:single
```

可视化模式：

```bash
npm run test:ui
```

## 7.3 当前测试覆盖重点

自动化测试已经覆盖以下核心流程：

- 页面基础加载
- 看板只读详情
- 看板任务持续时间展示
- 未分配任务显示
- 项目、成员、任务的增删改
- 项目状态与任务约束
- 会议创建、保存、删除、查询
- 导入参会人员
- 导入未完成任务
- 会议进度回写
- 会议历史任务详情展示
- HTML 报告导出

## 8. 常见维护操作

### 8.1 清空数据

可以直接使用：

- `chip-todo/clear-data.html`

或在浏览器控制台执行：

```js
localStorage.clear()
```

### 8.2 查看当前数据

```js
JSON.parse(localStorage.getItem('chip_todo_data') || '{}')
JSON.parse(localStorage.getItem('chip_todo_meetings') || '[]')
```

### 8.3 排查界面异常

建议按顺序检查：

1. 浏览器控制台是否有运行时错误
2. `localStorage` 中是否存在旧格式数据
3. `store.js` 的归一化逻辑是否兼容旧记录
4. 对应功能是否已有 Playwright 测试覆盖

## 9. 已知设计约束

- 所有数据仅保存在本地浏览器，没有多端同步
- 没有账户和权限体系
- 没有后端接口和数据库
- 没有实时协作能力

这些约束是当前设计取舍的一部分，换来的是部署简单、启动成本低、离线也能用。
